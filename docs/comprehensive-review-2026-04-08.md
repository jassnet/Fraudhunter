# FraudChecker 総合レビュー 2026-04-08

---

## A. エグゼクティブサマリー

### 全体評価

FraudChecker は、アフィリエイト不正検知のための社内ツールとして、**全体的に高い品質を持つ**プロダクトです。FastAPI + Next.js + PostgreSQL のモダンスタックを活用し、耐久性のあるジョブキュー、設定バージョニング、検出リネージュなど、運用を見据えた設計が随所に見られます。コードの可読性、テストカバレッジ、セキュリティ意識のいずれも水準以上です。

一方で、**セキュリティ上のタイミング攻撃リスク、パフォーマンスボトルネック（`CAST` による索引無効化）、console.py の肥大化**が主要な改善ポイントです。ビジネス観点では、MVP として十分な機能セットを備えていますが、エクスポート機能やアクセスログの可視化など、運用チームの生産性を直接向上させる改善余地があります。

### 最も重要な懸念トップ 5

1. **[Security/High] API キー比較にタイミング攻撃脆弱性** — `!=` 演算子による文字列比較（`api_dependencies.py:43-44`）
2. **[Performance/High] `CAST(conversion_time AS date)` による索引無効化** — console.py の複数クエリでフルスキャン発生リスク（`console.py:549, 688`）
3. **[Code Quality/Medium] console.py が 948 行と肥大化** — 複数の責務が混在し保守性が低下
4. **[Security/Medium] データセンター IP 判定が粗すぎる** — `"3."` プレフィックスは全 3.0.0.0/8 に一致し誤判定リスク大（`ip_filters.py:33-60`）
5. **[Operations/Medium] レート制限なし** — API エンドポイントに対する abuse 防御が未実装

### 良い点トップ 5

1. **耐久性のあるジョブキュー設計** — リース、ハートビート、リトライ、冪等キー、同時実行制御を備えた堅牢な実装（`job_status_queue.py`）
2. **検出リネージュと設定バージョニング** — 過去の検出結果が「どの設定・どのコードバージョンで生成されたか」追跡可能（`findings.py`, `settings_versions` テーブル）
3. **多層セキュリティ**: 環境ガード、読み取り専用/管理者の分離、本番での安全デフォルト強制（`runtime_guards.py`）
4. **テスト戦略**: ビジネスシナリオ駆動の命名（`_behavior.py`）、MSW によるフロントエンドテスト、24 ファイルのバックエンドテスト
5. **フロントエンド UI 設計**: レスポンシブ対応、ARIA ラベル、キーボードナビゲーション、日本語ローカライズ、アラートグルーピング

### 今すぐ直すべき点

- API キー比較を `hmac.compare_digest()` に置換
- `CAST(conversion_time AS date)` を生成列 + 索引に置換
- `_env_truthy` の重複定義を統一

### 今は見送ってよい点

- 検出リネージュの簡素化（現状で運用に問題なし）
- マルチテナント対応（現時点で不要）
- 国際化対応（社内ツールのため日本語固定で問題なし）

### C-1. コード品質

### C-4. セキュリティ

#### C-4-1. API キーのタイミング攻撃脆弱性

- **観点**: セキュリティ / 認証
- **重要度**: **High**
- **問題**: `api_dependencies.py:43` で `if token != expected:` による文字列比較を使用。Python の `!=` は先頭から比較し、最初に不一致が見つかった時点で終了するため、レスポンスタイムの差異からキーを推測可能
- **根拠**: `api_dependencies.py:43-44`, `api_dependencies.py:100`
- **影響**: 攻撃者が API キーを 1 文字ずつ特定可能（理論上）。実際のネットワーク遅延でノイズが乗るため即座に悪用は困難だが、セキュリティベストプラクティスに反する
- **推奨アクション**: `import hmac` → `hmac.compare_digest(token, expected)` に置換。全3箇所（admin, analyst, test_key）
- **トレードオフ**: なし（純粋な改善、パフォーマンス影響ゼロ）
- **実装コスト感**: S
- **優先度**: **P0**

#### C-4-2. DATACENTER_IP_PREFIXES が粗すぎる

- **観点**: セキュリティ / 検出精度
- **重要度**: **Medium**
- **問題**: `ip_filters.py:33-60` の `DATACENTER_IP_PREFIXES` で `"3."`, `"13."`, `"18."`, `"20."` 等のプレフィックスを使用。`"3."` は 3.0.0.0/8（約 1670 万 IP）全体に一致し、一般ユーザーの IP も含む。誤検知（false positive）のリスクが高い
- **根拠**: `ip_filters.py:34-39`
- **影響**: 不正でないコンバージョンが「データセンターIP」と判定され、ユーザー被害につながる可能性
- **推奨アクション**:
  1. AWS/GCP/Azure の公式 IP レンジリスト（JSON 形式で公開されている）を使用
  2. `ipaddress` モジュールの `ip_address in ip_network` を使用し、CIDR ベースでの判定に切り替え
  3. 短期的には、より詳細なプレフィックス（例: `"3.5."`, `"3.8."` 等 AWS 固有レンジ）に絞る
- **トレードオフ**: IP リストの定期更新が必要（月次程度）、メモリ使用量微増
- **実装コスト感**: M
- **優先度**: P1

#### C-4-3. レート制限なし

- **観点**: セキュリティ / Abuse 対策
- **重要度**: **Medium**
- **問題**: FastAPI エンドポイントにレート制限が見当たらない。API キーの総当たり攻撃や、リソース枯渇攻撃に対して無防備
- **根拠**: 全 API ルーターファイル、`app` 構成コード
- **影響**: DDoS や総当たり攻撃に対する耐性が低い
- **推奨アクション**:
  1. Render のプラットフォームレベルのレート制限（もし提供されていれば）を活用
  2. `slowapi` ライブラリによるアプリケーションレベルのレート制限
  3. 最低限、認証エンドポイントに対する IP ベースのレート制限
- **トレードオフ**: ライブラリ依存追加、正当なバッチ処理への影響
- **実装コスト感**: S
- **優先度**: P1

#### C-4-4. CORS 設定の確認不足

- **観点**: セキュリティ / CSRF
- **重要度**: Medium
- **問題**: `.env` に `FC_CORS_ORIGINS` が定義されているが、FastAPI の `CORSMiddleware` 設定コードが確認できない。フロントエンドプロキシ経由のみの通信であれば問題は軽減されるが、直接 API アクセスの場合のリスクが残る
- **根拠**: `.env:14`, FastAPI アプリ構成コード（要確認）
- **影響**: クロスオリジン攻撃のリスク
- **推奨アクション**: CORSMiddleware の設定コードを確認し、必要に応じて追加
- **トレードオフ**: なし
- **実装コスト感**: S
- **優先度**: P1（要確認）

#### C-4-5. `FC_ALLOW_INSECURE_ADMIN` の存在リスク

- **観点**: セキュリティ / セキュアデフォルト
- **重要度**: Low
- **問題**: `FC_ALLOW_INSECURE_ADMIN=true` で認証なしの管理者アクセスが可能。本番環境での有効化は `runtime_guards.py:59-60` でブロックされているが、ステージング環境等での有効化忘れリスク
- **根拠**: `api_dependencies.py:36-39`, `runtime_guards.py:59-60`
- **影響**: ステージング環境で認証なしのデータ改変が可能になるリスク
- **推奨アクション**: 起動時に WARNING ログを出力（現状もガードはあるが、dev/local 以外の環境で有効化された場合のログが不足）
- **トレードオフ**: なし
- **実装コスト感**: S
- **優先度**: P3

#### C-4-6. ログへの IP アドレス・ユーザーエージェント記録

- **観点**: セキュリティ / 個人情報
- **重要度**: Low
- **問題**: IP アドレスとユーザーエージェントは個人情報に該当しうる（GDPR 下では個人データ）。現状の日本国内運用では問題ないが、将来のグローバル展開時にリスク
- **根拠**: `suspicious_conversion_findings` テーブル、アラート詳細 API レスポンス
- **影響**: プライバシー規制への抵触リスク（現状は低い）
- **推奨アクション**: 現時点では問題なし。グローバル展開時にはマスキングやアクセス制御の強化を検討
- **トレードオフ**: マスキングは検出精度に影響する可能性
- **実装コスト感**: M（将来）
- **優先度**: P3

### C-5. パフォーマンス

#### C-5-1. `CAST(conversion_time AS date)` による索引無効化

- **観点**: パフォーマンス / DB クエリ
- **重要度**: **High**
- **問題**: `console.py` の複数クエリで `CAST(c.conversion_time AS date) = :target_date` を使用。この式は `conversion_time` カラムへの索引が使えず、テーブルフルスキャンになる
- **根拠**: `console.py:491`, `console.py:549-550`, `console.py:688`
- **影響**: `conversion_raw` テーブルの行数増加に伴い、ダッシュボード・アラート詳細の応答時間が線形に悪化
- **推奨アクション**:
  1. **Quick Win**: `WHERE c.conversion_time >= :target_date::date AND c.conversion_time < :target_date::date + INTERVAL '1 day'` に書き換え（既存索引が使える）
  2. **中期**: `conversion_date` 生成列を追加し、索引を作成
- **トレードオフ**: Quick Win はクエリが少し冗長になる。生成列はマイグレーション追加が必要
- **実装コスト感**: S（Quick Win）/ M（生成列）
- **優先度**: **P0**

#### C-5-2. ダッシュボードの N+1 的パターン

- **観点**: パフォーマンス / クエリ効率
- **重要度**: Medium
- **問題**: `get_dashboard()` が `_fetch_alert_rows` → `_fetch_alert_transaction_summary` → `_fetch_affiliate_conversion_totals` → `get_daily_stats` と 4 回以上の DB ラウンドトリップを実行
- **根拠**: `console.py:36-83`
- **影響**: ダッシュボード表示のレイテンシ増加（特にネットワーク遅延が大きい環境で顕著）
- **推奨アクション**:
  1. 一部クエリの統合（`_fetch_alert_rows` と `_fetch_alert_transaction_summary` を JOIN で統合）
  2. ダッシュボード結果の短時間キャッシュ（30-60 秒 TTL）
- **トレードオフ**: クエリ統合はクエリの複雑化、キャッシュは鮮度との兼ね合い
- **実装コスト感**: M
- **優先度**: P2

#### C-5-3. `_fetch_program_unit_prices` の全行 raw_payload 取得

- **観点**: パフォーマンス / データ転送量
- **重要度**: Medium
- **問題**: `_fetch_program_unit_prices`（`console.py:527-572`）が `raw_payload` 全体を取得し、Python 側で報酬額を抽出。payload が大きい場合（数 KB/行）、大量のデータ転送が発生
- **根拠**: `console.py:546-557`
- **影響**: メモリ使用量とネットワーク帯域の浪費
- **推奨アクション**: PostgreSQL の JSON 関数（`jsonb_extract_path_text`）を使用してDB側で抽出。または、報酬額を専用カラムに非正規化
- **トレードオフ**: DB 側の JSON 関数使用は DB 依存増、非正規化はスキーマ変更が必要
- **実装コスト感**: M
- **優先度**: P2

#### C-5-4. 計測すべきメトリクス提案

以下のメトリクスの計測を推奨:
- ダッシュボード API のレスポンスタイム（P50/P95/P99）
- アラート一覧 API のレスポンスタイム
- `conversion_raw` テーブルのサイズ推移
- ジョブキューの処理レイテンシ（enqueue → complete）
- DB コネクションプール使用率

### C-6. UI/UX

#### C-6-1. アラートの全文検索・フリーテキスト検索がない

- **観点**: UI/UX / 操作効率

## E. セキュリティレビュー要約

### 想定される脅威

1. **認証バイパス**: API キーの総当たり攻撃（タイミング攻撃併用）
2. **データ漏洩**: IP・UA の不正取得によるユーザー追跡
3. **サービス妨害**: レート制限なしを悪用した DoS
4. **不正操作**: admin キー漏洩時の全アラートステータス改変

### 攻撃面（Attack Surface）

| 攻撃面 | リスク | 備考 |
|---|---|---|
| FastAPI エンドポイント（8 ルーター） | Medium | 認証あり、レート制限なし |
| Next.js API Routes（5 ルート） | Low | サーバーサイドプロキシ、キー非露出 |
| ACS API 通信 | Low | HTTPS（推測）、サーバー間通信 |
| PostgreSQL | Low | ネットワーク分離（Render 内部） |
| CLI コマンド | Low | サーバー上での直接実行のみ |

### 重大なリスク

| リスク | 重大度 | 対策状況 |
|---|---|---|
| API キータイミング攻撃 | High | 未対策 |
| データセンター IP 誤判定 | Medium | 不十分 |
| レート制限なし | Medium | 未対策 |
| CORS 設定不明 | Medium | 要確認 |

### すぐに入れるべき防御

1. `hmac.compare_digest()` への置換（工数: 30 分）
2. CI への `pip-audit` / `npm audit` 追加（工数: 1 時間）
3. 認証エンドポイントのレート制限（工数: 2 時間）

### 追加で必要なテストや監査

- API キーのブルートフォース耐性テスト
- OWASP ZAP による動的スキャン
- 依存関係の CVE チェック自動化
